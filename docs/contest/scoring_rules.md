# 华为企业赛题三评分规则文档

---

## 1. 评分总览

赛题三总成绩由两部分组成：

| 组成部分 | 权重 | 评测方式 |
|---|---:|---|
| 竞赛得分 | 60% | 公开测试集 30% + 隐藏测试集 30% 自动评分 |
| 专家评分 | 40% | 创新性 15% + 方案落地可行性 15% + 算法泛化性 10% |

---

## 2. 竞赛得分

竞赛得分占总成绩 60%。

其中：

| 子项 | 权重 |
|---|---:|
| 公开测试集自动评分 | 30% |
| 隐藏测试集自动评分 | 30% |

公开测试集用于开发调试和阶段性验证；隐藏测试集用于检验泛化能力，不能依赖对公开数据的过拟合。

---

## 3. 自动评分指标体系

赛题文档给出的自动评测指标如下：

| 指标 | 权重 | 说明 |
|---|---:|---|
| F1 Score | 70% | 综合衡量精确率与召回率的平衡 |
| 运行效率 | 20% | API 调用次数、Token 消耗量、端到端延时 |
| 回复结果结构化 | 10% | 返回结果是否使用列表、关系图等结构化形式 |

---

## 4. F1 Score

### 4.1 指标权重

F1 Score 占自动评分的 70%，是最重要的指标。

### 4.2 指标含义

F1 Score 用于综合衡量 Precision 和 Recall：

- Precision：系统返回的论文中，有多少是真正相关的；
- Recall：所有应返回的相关论文中，系统找回了多少；
- F1：Precision 和 Recall 的调和平均值。

### 4.3 标准公式

```text
Precision = TP / (TP + FP)

Recall = TP / (TP + FN)

F1 = 2 * Precision * Recall / (Precision + Recall)
```

其中：

- TP：系统返回且真实相关的论文；
- FP：系统返回但真实不相关的论文；
- FN：系统未返回但真实相关的论文。

### 4.4 对系统设计的影响

为了提升 F1，系统需要同时优化：

1. 高召回检索；
2. 低噪声过滤；
3. 精细相关性判断；
4. 稳定重排序；
5. 多源结果去重；
6. 查询分解与扩展；
7. 引文网络扩展；
8. 避免返回大量低质量论文。

### 4.5 建议工程指标

在本项目中建议记录：

```json
{
  "precision_at_20": 0.0,
  "recall_at_20": 0.0,
  "f1_at_20": 0.0,
  "recall_at_5": 0.0,
  "recall_at_10": 0.0,
  "recall_at_50": 0.0
}
```

---

## 5. 运行效率

### 5.1 指标权重

运行效率占自动评分的 20%。

### 5.2 官方说明

运行效率主要考察：

1. API 调用次数；
2. Token 消耗量；
3. 端到端延时。

### 5.3 对系统设计的影响

系统不能为了提高召回率无限制搜索，也不能对每一篇候选论文都调用昂贵 LLM。

需要设计：

- 检索预算；
- LLM 调用预算；
- Token 预算；
- 缓存机制；
- early stop；
- 批量判断；
- fallback 规则模型；
- 多阶段过滤；
- 低成本初筛 + 高成本精排。

### 5.4 建议记录字段

```json
{
  "api_call_count": 0,
  "search_api_call_count": 0,
  "llm_call_count": 0,
  "estimated_input_tokens": 0,
  "estimated_output_tokens": 0,
  "estimated_total_tokens": 0,
  "latency_seconds": 0.0,
  "cache_hit_count": 0,
  "search_rounds": 0,
  "judged_paper_count": 0
}
```

### 5.5 建议策略

#### 5.5.1 分阶段过滤

```text
多源检索
  ↓
规则初筛
  ↓
去重
  ↓
轻量相关性判断
  ↓
只对 Top-N 调用 LLM
  ↓
重排序
```

#### 5.5.2 搜索轮数控制

建议默认：

- 初始检索轮数：1；
- Query Evolution 轮数：1；
- RefChain 深度：1；
- 最大候选论文数：100-300；
- 进入 LLM 精判的论文数：20-50。

#### 5.5.3 缓存

至少缓存：

- 搜索 API 返回；
- 论文详情；
- relevance judgement；
- reranking 结果；
- LLM query understanding 输出。

---

## 6. 回复结果结构化

### 6.1 指标权重

回复结果结构化占自动评分的 10%。

### 6.2 官方说明

返回结果是否使用列表、关系图等结构化展示形式。

### 6.3 推荐输出结构

Markdown 输出建议包含：

```markdown
# 查询理解

# 检索策略

# 高度相关论文

# 部分相关论文

# 方法路线归纳

# 时间线

# 引文或关系图

# 证据不足与不确定信息

# 成本与效率统计
```

JSON 输出建议包含：

```json
{
  "query_analysis": {},
  "search_plan": {},
  "highly_relevant_papers": [],
  "partially_relevant_papers": [],
  "method_clusters": [],
  "timeline": [],
  "citation_graph": {},
  "missing_evidence": [],
  "cost_report": {}
}
```

### 6.4 结构化输出验收标准

- [ ] 能区分高度相关和部分相关；
- [ ] 每篇论文有标题、作者、年份、venue、链接、摘要或理由；
- [ ] 每篇论文有相关性解释；
- [ ] 有方法类别或主题类别；
- [ ] 有时间线或研究进展；
- [ ] 有证据不足说明；
- [ ] 有成本统计；
- [ ] Markdown 可读；
- [ ] JSON 可解析。

## 8. 创新性

### 8.1 权重

创新性占总成绩 15%。

### 8.2 可体现的创新点

本项目可从以下方向体现创新性：

1. 面向复杂学术查询的结构化 Query Understanding；
2. 多源学术检索融合；
3. 单层 RefChain 引文扩展；
4. Query Evolution 检索策略迭代；
5. 相关性判断与证据解释联合建模；
6. 面向 F1 的高召回与高精度平衡策略；
7. 成本约束下的动态 early stop；
8. 结构化结果归纳；
9. 支持证据不足和不确定性输出；
10. 面向不同学科的检索源选择策略。

### 8.3 不建议主打的创新点

不建议在 MVP 阶段主打：

- 自研大模型；
- 大规模 RL 训练；
- 全文向量库从零构建；
- 复杂知识图谱；
- 过重 Web UI；
- 端到端全自动科研发现。

这些方向开发成本高，且不一定直接提升赛题核心指标。

## 9. 方案落地可行性

### 9.1 权重

方案落地可行性占总成绩 15%。

### 9.2 评委可能关注的问题

- 系统是否能运行；
- 是否能复现；
- API Key 是否可配置；
- 是否支持无 Key 降级；
- 是否有清晰 README；
- 是否有测试脚本；
- 是否能跑公开评测；
- 是否有日志和错误处理；
- 是否能控制成本；
- 是否能输出结构化结果。

### 9.3 工程落地要求

建议提交材料中包含：

1. 代码；
2. 运行说明；
3. 环境配置；
4. API 配置说明；
5. Demo 示例；
6. 公开测试集结果；
7. 错误处理说明；
8. 成本统计；
9. 模块设计文档；
10. 消融实验设计。

## 10. 算法泛化性

### 10.1 权重

算法泛化性占总成绩 10%。

### 10.2 风险

隐藏测试集可能包含：

- 不同学科；
- 不同查询风格；
- 中文查询；
- 英文查询；
- 宽泛查询；
- 细粒度查询；
- 时间约束；
- venue 约束；
- 方法比较；
- 数据集检索；
- 应用型检索；
- 综述型检索。

### 10.3 泛化优化策略

建议支持：

1. 中英文查询；
2. 多 intent 类型；
3. 多检索源；
4. query rewriting；
5. source selection；
6. domain detection；
7. 规则 fallback；
8. LLM fallback；
9. 缓存；
10. 结果去重；
11. 不确定性输出。

## 11. 内部开发评分目标

建议项目内部以如下目标推进：

| 阶段 | 目标 |
|---|---|
| MVP | 能跑通完整 pipeline |
| Alpha | 能在 LitSearch 子集上输出 Recall@5/20 |
| Beta | 加入 Query Evolution、RefChain、Reranker |
| Final | 完成报告、消融实验、成本统计、Demo |

---

## 12. 实验建议

至少设计以下实验：

### 12.1 主实验

对比：

1. 仅关键词检索；
2. 多源检索；
3. 多源检索 + 去重；
4. 多源检索 + Judgement；
5. 多源检索 + Judgement + Reranker；
6. 完整系统。

### 12.2 消融实验

建议消融：

- 不使用 Query Understanding；
- 不使用 Query Evolution；
- 不使用 RefChain；
- 不使用 LLM Judgement；
- 不使用 Reranker；
- 不使用缓存；
- 不使用多源检索。

### 12.3 效率实验

记录：

- 平均 API 调用次数；
- 平均 LLM 调用次数；
- 平均 Token 消耗；
- 平均端到端耗时；
- 平均候选论文数量；
- 平均最终返回论文数量。

## 14. 开发注意事项

1. 不要只做普通 RAG；
2. 不要只返回论文标题列表；
3. 不要忽视成本统计；
4. 不要对公开集过拟合；
5. 不要无限制调用 LLM；
6. 不要忽视隐藏测试集泛化；
7. 不要写死某个领域；
8. 不要依赖不可复现的本地环境；
9. 不要硬编码 API Key；
10. 不要夸大实验结果。

---

## 15. 评分导向下的优先级

| 优先级 | 模块 | 原因 |
|---|---|---|
| P0 | 多源检索 + 去重 | 直接影响 Recall |
| P0 | Judgement + Reranker | 直接影响 Precision 和 F1 |
| P0 | 结构化输出 | 直接影响 10% 结构化得分 |
| P1 | Query Understanding | 提升复杂查询处理能力 |
| P1 | Query Evolution | 提升召回与查询覆盖 |
| P1 | RefChain | 提升深层相关论文召回 |
| P1 | 成本统计 | 影响效率评分 |
| P2 | Web UI | 展示加分，但不是核心 |
| P2 | 全文 RAG | 有帮助，但成本较高 |
| P3 | RL 训练 | MVP 阶段不建议 |

---

## 16. 推荐最终系统目标

最终系统应该做到：

```text
输入一个复杂学术查询后，系统可以：
1. 解释用户查询意图；
2. 拆分多个检索子问题；
3. 调用至少一种学术搜索 API；
4. 返回候选论文；
5. 过滤不相关论文；
6. 通过排序区分高相关和部分相关；
7. 给出每篇论文为什么相关；
8. 按方法、时间、主题归纳结果；
9. 记录成本、Token 和耗时；
10. 输出 Markdown 和 JSON。
```
